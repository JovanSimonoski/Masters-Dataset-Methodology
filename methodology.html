Cybersecurity Attacks Dataset

Methodology for creating a labeled network attack dataset using intentionally vulnerable VMs and volunteer attackers.

01 General Info

This project is focused on creation of a Cybersecurity Attacks Dataset which can later be used for IDS/IPS systems, ML, or any other relevant purpose.

The idea is to have an isolated environment with intentionally vulnerable VMs and volunteer attackers who will conduct the attacks. The following sections cover each aspect of the project and form the complete methodology.

02 Scenario

A controlled environment with vulnerable machines and attacker machines.

Vulnerable Machines

7 instances of each type:

  • Metasploitable 2 VMs - Linux
  • Metasploitable 3 VMs - Windows
  • SecGen Custom VMs - still under testing
  • VulnHub Docker Containers - recent CVEs, different container combinations per VM

Attacker Machines

Each attacker is given a preconfigured Kali Linux VM inside the environment with all required tools. Total: 7 Kali Linux VMs.

03 Monitoring

RAW Packet Capture

Still to be tested - waiting on Bidik's proposal.

NTP Synchronization

  • NTP server: ntp.finki.ukim.mk - configured on all machines
  • Linux (systemd): /etc/systemd/timesyncd.conf
  • Older Linux: manual NTP installation
  • Windows: Windows Time Service (W32Time)

Logs

  • VictoriaLogs - stores all system logs, pulls from Kafka queue
  • Log Agent on each machine - ships logs to Kafka. Options: Grafana Promtail, Fluent Bit
  • Kafka - receives and queues logs from each machine

NetFlow

  • nprobe - NetFlow probe on the firewall
  • ntopng - NetFlow web interface
  • Kafka - optional queuing of NetFlow packets from nprobe

Attack Outcomes

success - exploited successfully
ran - executed without error, no exploit (also: scanning)
error - tool errored / attack failed

05 Automatic Labeling

Dedicated to automatic labeling of flows and raw packet captures after conducting attacks. A CLI tool called attacklog - written in pure Python - records attack start and end events into a local SQLite database.

Process

Each attacker runs attacklog (pre-installed on their Kali VM) to bracket every attack. This produces one structured record per attack, joined against captured flows and packets during post-processing.

What Each Record Captures

FieldDescription
attack_idUnique UUID - auto-generated per attempt
attacker_id + src_ipSet once at init. Format: attacker_<number>
attack_nameCanonical name matching the Attacks section
dst_ip + dst_port + protocolEntered at start
t_start / t_endMillisecond-precision UTC timestamps from system clock
statussuccess, ran, or error
notesOptional free-text field for anomalies

Attacker Workflow (per attack)

  1. Run attacklog start with attack name, destination IP, port, and protocol
  2. Execute the attack following the step-by-step instructions
  3. Run attacklog end with the outcome status

Setup

  • Each attacker receives a preconfigured Kali VM
  • NTP synced to ntp.finki.ukim.mk on all machines before any session - critical for timestamp joining
  • attacklog is pre-installed and initialized with the attacker's attacker_id and static IP
  • Static IP per attacker makes src_ip a reliable identifier during flow/packet matching

Labeling Logic

The labeling script joins the attacklog export against NetFlow and PCAP data using:

  • src_ip + dst_ip + dst_port + protocol as the join key (source port recovered from PCAP)
  • t_start / t_end as the time window with ±1s jitter tolerance

Any flow outside a matched window is labeled benign. Safe in this isolated environment - no legitimate traffic exists on target service ports.

Open Attack Guard

attacklog refuses to start a new attack if a previous one has no t_end.

Data Storage & Export

  • Records stored at ~/.attacklog/attacks.db (SQLite)
  • Exported via attacklog export --format json or --format csv at session end
  • Kafka integration deferred - to be discussed

Not Recorded (by design)

  • Callback sessions (dst → src) - recoverable from PCAP if needed
  • CVE, tool, payload - defined in attack files, joined by attack_name during post-processing

Tool Reference

Initialization
python attacklog.py init --attacker-id attacker_<number> --src-ip <src_ip>
Start attack
python attacklog.py start --name <attack_name> --dst-ip <dst-ip> --dst-port <port>
End attack
python attacklog.py end --status <success|ran|error> --notes <optional>
List attacks
python attacklog.py list
Export
python attacklog.py export --format <json|csv> --output <file>
Add to PATH
chmod +x attacklog.py
sudo mv attacklog.py /usr/local/bin/attacklog
attacklog init --attacker-id attacker_<number> --src-ip <src_ip>
